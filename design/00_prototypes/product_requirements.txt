MAO 营销多智能体协同编排平台 - 全景架构与产品设计总集 (V5.0 终极完整版)
第一部分：产品需求文档 (PRD)
1.1 产品愿景与定位
MAO 平台定位为赋能全业务线的**“AI 智能体编排底座”与“统一运营工作台”** 1。其核心理念是：“业务逻辑归业务系统，AI 调度与可视化编排归 MAO 平台” 1。平台通过底层的标准化接入和低代码组装，在前端通过自然语言 (LUI) 彻底释放业务生产力 1。
1.2 核心业务模块设计
平台分为面向日常使用的 C 端工作站，以及面向架构与管理的 B 端控制台 1。
C端：LUI 统一工作站 (User Workspace)
LUI + GUI 融合交互： 接收自然语言意图并流式输出，当涉及参数确认、高危写入时，前端将动态渲染可交互的表单卡片 2。卡片提交后即锁定为不可交互的灰色只读状态 2, 3。
AI 托管大盘 (My Scheduler)： 侧边栏独立展示个人名下的后台定时任务和盯盘任务，支持直观查看存活期、一键暂停/恢复、强制终止以及过期续签 2。
B端：MAO 管理控制台 (Management Console)
智能体工厂 (Agent Factory)： 可视化组装领域 Agent，支持 System Prompt 编写（如明确要求“严禁捏造虚假数据，必须客观简洁输出 Markdown”）、RAG 知识库关联与专属 Tool（如 RunRiskCheckSOP、QueryTaskConfig）复选框挂载绑定 2, 4。
全局编排画布 (Workflow Canvas)： 基于底层图表库（如 React Flow）的 DAG 可视化界面，用于将高危、长链路业务固化为标准操作程序 (SOP) 2。
技能注册中心 (Skill Registry)： 统一导入与管理底层 OpenAPI 和视图/异步组件，提供 4 种底层技能类型（API、View、Async、Macro）的快速注册向导，并配置鉴权与拦截规则 2, 5-8。
定时调度管理与会话审计 (Trace Logs)： 管理全局 Cron 任务，提供 100% 可观测的“链路脑电图”，精准记录路由、推演及工具调用的 JSON 级明细 2。
1.3 核心交互体系与人机协作 (HITL) 体验
“卡片即技能”与轻量级确认： 大模型绝不直接生成 UI 代码，而是输出 JSON 参数调用预设的“视图技能 (View Skill)” 9, 10。前端 BFF 拦截后渲染高保真表单，服务端进入“轻量级挂起 (Interactive Pause)”等待用户点击提交 11, 12。
Session 与 Task 的 1:N 多道复用： 会话视窗 (Session) 仅为前端展示容器，底层可并行跑多个任务 (Task) 13。当某任务触发长程挂起时，引擎越权向前端推送“等待审批卡片”，前端输入框立即解锁 14, 15。
意图消歧 (Intent Disambiguation)： 任务挂起期间，用户若发起新对话，Router 大脑会自动结合当前活跃任务列表进行消歧 16, 17。若用户输入“撤回刚才的审批”，则干预旧任务；若输入无关指令，则并发孵化物理隔离的新 Task 16, 17。
离线信箱与主动触达闭环 (Offline Inbox)： 异步任务在后台完成后，若用户在线，则通过 WebSocket 主动推送并带上“引用锚点 (Quote Anchor)”（如“🔗 引用：配置确认单 -> 审批已通过”） 17, 18。若用户离线，消息将落入“离线信箱”；次日登录时，UI 右上角铃铛图标亮起红点，聊天流底部补绘消息卡片，并加盖一条浅灰色的“———— 以下为离线期间执行完毕的任务 ————”视觉分割线 10, 19-21。
会话记忆温和管控： 为防 Token 爆仓，后端采用滑动窗口压缩记忆 22。当触发深度压缩时，前端聊天框顶部需轻量弹出“💡 为保证响应速度与调度性能，过往长时记忆已自动归档”的提示，并强化顶部的【清空重置】按钮视觉，引导用户养成重置习惯 10, 23, 24。
第二部分：全局架构设计 (HLD)
2.1 全局系统架构分层 (6-Layer Architecture)
MAO 采用**“控制面与执行面彻底分离”**的设计，全景分为六层边界 25-27：
客户端体验层： C端融合工作站 (LUI+GUI) 与 B端管理控制台 26, 27。
统一接入网关层： BFF API 网关（处理 WS/HTTP 推流与离线信箱）和 统一事件网关（抹平内部 Kafka MQ 订阅与外部 HTTP Webhook 回调） 26, 27。
调度控制面 (大脑)： 包含 Router 路由、Session/Task 并发管理器以及存储深冻结快照与幂等防线的 StateDB (Redis/DB) 26, 27。
智能执行面 (引擎)： 包含 ReAct Runner (Agent 引擎)、DAG Runner (画布引擎)、带 Zod 校验的共享黑板机制 (Blackboard)，以及执行强拦截的框架工具执行器 (Tool Executor) 26, 27。
资产与中枢层： 技能注册中心与 RAG 向量知识库 26, 27。
外部生态层： 核心业务微服务 (秒级返回)、OA 审批系统 (长程依赖) 及 Kafka 消息队列 26, 27。
2.2 控制面设计：意图路由与会话管理
两级分发架构 (Two-Level Dispatching)： 全局 Router 绝对不直接分发具体的“工具 (Skill)” 25, 28。Router 的匹配依据是载体的“职责边界（Description）”，只负责将意图分发给特定的 Agent 或 SOP 画布 29。被唤醒的 Agent 在内部推演时，才自主决定挑选兜里挂载的哪项工具 29。
交互式消歧兜底： 当多个 Agent 的职责出现重叠，导致 Router 计算出的置信度极其接近（如 Agent A 85分，Agent B 83分），或完全无法匹配时，Router 绝不盲猜，而是中止分发，在前端抛出【意图澄清卡片】，由人类手动点选承接方 30, 31。
2.3 执行面设计：双引擎与多技能协同
Agent 内部的三大协同模式：
串行推演 (Sequential ReAct)： 面向互有依赖的技能调用，如获取 UserID 后再调用发券接口 32。
并行并发调用 (Parallel Tool Calling)： 面向无关联的信息收集。利用现代大模型能力，Agent 可输出包含多个指令的 Array，底层工具执行器开启多线程 (Goroutine/Promise.all) 并发请求，打包一次性返回，极大地降低网络延迟与 Token 消耗 33, 34。
降维打击 (SOP Macro Skill)： 面向包含 5 个以上步骤的高危长链路，严禁大模型盲猜流转，而是强制调用由架构师在画布画好的“宏工具”，将执行权彻底移交给确定的 DAG 引擎 34。
双引擎无限嵌套： ReAct 引擎与 DAG 画布引擎可互相递归调用（Agent in Workflow, Workflow in Agent），实现高度的混合编排能力 35-37。
强类型共享黑板 (Typed Blackboard)： 在 DAG 节点流转中彻底废弃易导致幻觉的全量对话透传 38。所有 Agent 通过内存中的全局哈希表（黑板）进行数据握手 37。
强结构约束： 黑板配备预定义的 JSON Schema，Agent 写入时利用 Zod/Pydantic 引擎进行拦截校验 37, 38。并在系统 Prompt 末尾动态注入明确的 Markdown 约束指引（如“严禁创造任何其他键名”） 39。
参数精确映射： Agent 间解耦变量认知，利用连线侧边栏的参数映射面板 (Data Mapper) 进行 JSONPath 精确赋值提取 37, 39。
2.4 智能体技能树规范 (Skill Taxonomy)
对大模型而言心智负担为 0，均视为标准工具指令输出，但底层“工具执行器”实施多态路由分发 (Polymorphic Dispatcher)，严格分为四大类 40, 41：
原子接口 (API Skill)： 同步读写，毫秒级网络响应，即刻注入 Observation 40。
视图交互 (View Skill)： 阻断服务端推理流转，转交 BFF 驱动 WebSocket 推送前端渲染卡片，执行轻量级挂起 12。
长程异步 (Async Skill)： 声明式打上 x-mao-suspend: true 标签，触发大模型线程阻断与“状态机深冻结”，序列化落盘 Redis 等待外部回调唤醒 12。
复合编排 (Macro Skill)： 将完整 DAG 打包为单一工具启动，移交引擎内部调度嵌套 41。
第三部分：底层工程与安全容灾规范 (LLD)
3.1 异步流转与长程挂起机制
对于涉及审批的长程异步流转，必须执行框架强接管的 四阶段流转标准 15：
触碰边界： 引擎向外部 OA 发起请求（如 RunCompensateSOP），拿到 PENDING 凭证（如 {"oa_ticket_id": "OA-2024-991"}） 15, 42。
平滑回复： 引擎越权向前端直接推送“已挂起，等待审批”卡片，当前同步交互结束 15。
深冻结 (Deep Freeze)： 将任务内存状态完全序列化至 Redis/DB 中，释放线程与计算资源 14, 15。
回调唤醒与推送： 平台兼容 HTTP Webhook 与 高并发 Kafka MQ 订阅，并由统一事件网关清洗为标准 TaskResumeEvent 43。反序列化解冻后，将结果（如 APPROVED）伪装成 Observation 注入大模型唤醒执行，最终向 C 端发起 WebSocket 主动推送 15, 43。
僵尸兜底： 异步技能必须配置最大存活时间 (TTL，如 72 小时)。超期未回调则由调度器强制强杀标记为 Timeout_Killed 7, 43。
3.2 权限与身份安全防线
防 Cron 离线越权： 严禁完全无人值守的后台任务直接使用超级管理员权限 (God Token) 24, 44。
用户敏感操作： 托管限定 Scope 的 OAuth2 Refresh Token，每次触发前动态换取 5 分钟有效期的短期 Access_Token 执行 45。若 IAM 吊销权限，强杀任务为 Failed_Auth_Revoked 46。
微服务互信： 使用内部 AK/SK 鉴权时，强制在 HTTP Header 注入 X-Impersonate-User 模拟真实操作人信息，保证审计真实性 46, 47。
高危动作降级： 涉及资金划拨的高危 API 严禁在 Cron 中直接执行，必须自动降级生成审批单转交人工 24, 46。
PII 数据脱敏防线： 在工具执行器与大模型通信的网关层架设拦截器，进入模型前强制脱敏掩码真实手机号（如替换为 [PHONE_MASKED_01]），大模型输出调用工具时再由网关还原，确保模型不碰明文隐私 47, 48。
3.3 并发与容灾兜底防线
并发冲突与分布式实体锁： 针对 1:N 架构中多个 Task 同时操作同一业务实体（如同一活动 ID），底层引入 Redis 实体级互斥锁。命中锁定实体时强制拦截报错以防数据覆写死锁 21, 49。
大模型 Token 熔断机制： 设定全局最大推演步数 (Max_Steps，如 7 步)。当 ReAct 连续 3 次遭遇 Schema 校验失败或报错时，强制中断推演，向下发兜底卡片提示已保护系统资源 47, 50。
DAG 断点续传机制： 若 SOP 画布节点中途遭遇微服务宕机 (如 502)，保存执行断点快照。管理员可在审计日志中一键选择【从失败节点重试】或【跳过该节点】，严防重复跑全链路导致资损 47, 50。
脏数据重放与接口幂等性 (Idempotency)： 卡片流转状态（如变为灰色 Disabled）由后端 State 绝对强控，不受浏览器刷新影响 3, 23。C 端产生的系统级指令执行强制绑定 Task_ID + Card_Action_ID 作为全局唯一的幂等键。任何刷新或抓包重入的重复请求直接在网关层被丢弃，物理层面杜绝重复发奖等双花风险 (Double Spending) 3, 47, 51。
3.4 编排版本与状态机异常防线
全链路执行图快照 (Graph Snapshot)： 当 Task 实例化时，强制深度克隆其绑定的完整 SOP 画布结构快照。即使全局模板从 v1.0 更新至 v5.0，挂起的任务被唤醒时也仅按“出生时”快照执行 20, 24。审计日志必须打上版本标签（如 [执行版本: v1.0]），确保合规追溯拥有 100% 的脑电图一致性还原 22, 24, 47。
多智能体幻觉纠偏与必填项降级： 如果 Agent 无法输出符合黑板 Schema 的参数，先触发将“Schema 结构与错误输出打包”重新喂给模型的 Local Retry 37, 38。若 3 次局部重试仍无法输出下游必备的强依赖参数，执行引擎立即阻断流程，标记为 Failed_Data_Missing，并抛出异常卡片（如“缺失关键参数 user_id”），将机器死锁降级为向人类索要填参的轻量级交互 52, 53。
